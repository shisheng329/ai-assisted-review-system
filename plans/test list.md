# AI 驱动的文献筛选与分析系统

一个面向范围综述、系统综述与主题分析场景的多用户、多项目研究平台。系统围绕“导入题录数据、配置筛选标准、执行 AI 辅助筛选、开展 BERTopic 聚类分析、从 PDF 全文提取结构化信息”这条主线设计，目标是让用户尽量在一个项目空间内完成完整研究流程。

本 test list 不是简单的安装说明，而是一份面向 `20260423-system-demo-git-clean` 的“目标系统手册 + 交互规范参考文档”：

1. 对最终用户，它说明每个页面的作用、进入条件、具体操作方式和结果产出。
2. 对后续开发者，它定义系统正确时应该呈现的页面逻辑、交互要求和流程约束，方便作为后续修改代码的对照基准。

---

## 1. 系统简介

## 1.1 系统解决的问题

文献研究过程通常包含多个彼此关联但常被分散处理的步骤：

1. 整理和清洗题录数据。
2. 根据纳排标准进行标题与摘要初筛。
3. 对纳入文献做主题聚类和主题概览。
4. 从 PDF 全文中抽取结构化变量。
5. 保存中间状态、反复修订 Prompt、导出结果表。

本系统将这些步骤统一放在“项目”之下，减少在 Excel、脚本、聊天工具、文献软件之间来回切换。

## 1.2 适用对象

适合以下用户：

1. 进行范围综述、系统综述、证据图谱、文献计量辅助分析的研究生和研究人员。
2. 需要多人协作推进同一研究项目的课题组。
3. 需要对筛选标准、Prompt、主题命名和 PDF 提取模板进行持续迭代的用户。

## 1.3 核心能力

系统核心能力包括：

1. 多用户账户与多项目管理。
2. 文献题录数据导入、文件管理和当前数据源切换。
3. AI 辅助文献筛选。
4. Prompt 组装、双语审阅与版本历史。
5. BERTopic 聚类分析与图表展示。
6. 基于示例模板的 PDF 结构化提取。
7. 结果可视化、结果持久化和表格导出。

## 1.4 研究工作流总览

一个标准项目通常按如下顺序使用：

1. 注册并登录系统。
2. 在个人资料中配置 API。
3. 创建项目。
4. 在“数据接入”上传题录数据。
5. 在“文献筛选”设置标准、组装 Prompt 并执行筛选。
6. 在“聚类分析”对筛选结果做 BERTopic 分析。
7. 在“PDF 提取”基于模板提取全文信息。
8. 导出筛选结果、聚类结果和 PDF 提取结果。

## 2. 系统使用总流程

## 2.1 第一次使用

首次使用时，建议按下面顺序完成：

1. 进入系统首页。
2. 注册账户。
3. 登录账户。
4. 打开个人资料页面。
5. 配置可用的 API Key、Base URL 和默认模型。
6. 返回项目列表页。
7. 创建项目并进入项目工作区。

## 2.2 每次使用的典型顺序

常规使用时，建议按以下顺序推进：

1. 打开项目。
2. 进入“数据接入”检查当前数据文件是否正确。
3. 进入“文献筛选”配置标准或继续上次工作。
4. 完成筛选后视需要进入“聚类分析”。
5. 如果需要抽取全文变量，再进入“PDF 提取”。

## 2.3 一个项目中的推荐顺序

推荐的页面访问顺序：

1. 项目仪表盘：看总体进度。
2. 数据接入：确认数据源。
3. 文献筛选：完成标题与摘要筛选。
4. 聚类分析：对筛选结果做主题分析。
5. PDF 提取：对全文做结构化抽取。

## 3. 页面级详细操作手册

## 3.1 登录/注册页

### 页面作用

该页面用于完成：

1. 登录已有账户。
2. 注册新账户。
3. 切换系统语言。

### 主要区域

页面通常包含以下区域：

1. 系统标题区。
2. 登录表单。
3. 注册表单。
4. 语言切换控件。

### 如何使用

登录：

1. 输入用户名。
2. 输入密码。
3. 点击登录。
4. 登录成功后进入项目列表页。

注册：

1. 输入用户名。
2. 输入邮箱。
3. 输入密码。
4. 输入姓名或其他基础信息。
5. 提交注册。
6. 注册成功后返回登录并进入系统。

### 正确行为要求

1. 登录成功后，页面应跳转到项目列表页，而不是停留在登录页。
2. 登录状态在页面刷新后应保持有效。
3. 切换项目内其他功能时，不应因为 rerun 而回到登录页。
4. 语言切换后，登录相关文案应全部切换，不应出现 key 名或中英文混杂。

## 3.2 项目列表页

### 页面作用

项目列表页用于：

1. 展示当前账户下的所有项目。
2. 创建新项目。
3. 打开某个已有项目。

### 主要区域

1. 欢迎信息。
2. 项目列表区。
3. 创建项目入口。

### 如何使用

查看已有项目：

1. 页面按项目逐行展示项目名称、描述、创建时间等信息。

打开项目：

1. 点击该项目右侧的“打开项目”按钮。
2. 系统应直接进入对应项目。

创建项目：

1. 点击“创建项目”。
2. 填写项目名称。
3. 可选填写项目描述。
4. 提交后进入新项目。

### 正确行为要求

1. “打开项目”按钮必须能够直接进入对应项目，不能依赖用户再去左侧下拉选择。
2. 打开项目后应进入该项目的工作区，而不是回到系统主首页。
3. 当项目较多时，项目列表页仍应可快速定位项目。

## 3.3 个人资料 / API 配置页

### 页面作用

该页面用于：

1. 查看当前用户信息。
2. 配置默认 LLM API。
3. 保存后续筛选、双语审阅、PDF 提取和 LLM 后处理所需的模型参数。

### 基础配置项

至少应支持以下信息：

1. API Key。
2. Base URL。
3. 默认模型。
4. 语言偏好。

### 目标系统形态

从交互设计上，系统希望支持更高级的“API Key 库”管理方式：

1. 用户可保存多个供应商配置。
2. API Key 与模型/供应商绑定。
3. 支持常见国产模型供应商。
4. 刷新网页或重新登录后配置仍然保留。

### 如何使用

单套配置方式下：

1. 打开个人资料页。
2. 填写 API Key。
3. 填写 Base URL。
4. 填写默认模型。
5. 保存配置。

多配置方式目标形态下：

1. 选择已有供应商配置。
2. 或新增供应商配置。
3. 保存后设为当前生效配置。

### 正确行为要求

1. 若系统采用单套配置，则保存后应可被所有 LLM 功能复用。
2. 若系统采用 API Key 库，则用户应能清楚看到当前生效配置。
3. 用户配置的 API 不应在每次刷新后丢失。
4. 若系统要求“进入项目之前必须配置 API”，应在项目打开前给出明确提示，而不是进入项目后才报错。

## 3.4 项目仪表盘

### 页面作用

项目仪表盘用于提供项目总体概况，帮助用户快速判断当前研究进度。

### 主要指标

通常应展示：

1. 文献总量。
2. 已纳入数量。
3. 已生成主题数。
4. PDF 提取结果数。
5. 筛选结果分布图。
6. 最近活动时间。

### 如何使用

1. 进入项目后先查看项目仪表盘。
2. 根据文献数、纳入数和主题数判断当前项目处于哪个阶段。
3. 如果筛选尚未进行或聚类结果为空，可据此继续后续步骤。

### 正确行为要求

1. 仪表盘应只展示当前项目数据，不能混入其他项目数据。
2. 指标为 0 时应清楚提示“当前暂无数据”，而不是空白或报错。

## 3.5 数据接入页

数据接入页是项目内所有后续步骤的数据入口。

### 3.5.1 页面作用

该页面用于：

1. 上传文献题录文件。
2. 校验数据格式。
3. 预览数据。
4. 将数据保存到项目。
5. 管理项目内的多个数据文件。
6. 指定当前使用的数据源。

### 3.5.2 支持的输入文件

支持上传：

1. CSV
2. XLSX
3. XLS

### 3.5.3 最低字段要求

文献数据至少应包含：

1. `Record-id`
2. `Title`
3. `Abstract`

建议：

1. 一行对应一篇文献。
2. `Record-id` 尽量唯一。
3. 标题和摘要尽量不要为空。

### 3.5.4 页面区域说明

#### API 状态摘要区

作用：

1. 显示当前 API 配置是否可用。
2. 让用户知道后续筛选或提取是否具备调用条件。

#### 文件上传区

使用方式：

1. 上传 CSV 或 Excel 文件。
2. 系统读取文件并进行列校验。
3. 校验通过后展示预览。
4. 点击“保存到项目”。

正确行为：

1. 若缺少必要列，应直接提示缺失列名。
2. 校验通过后应提示数据可保存。
3. 点击保存后，应把文件和解析后的记录都存入项目。

#### 数据预览区

作用：

1. 让用户在保存前确认列名和内容是否正确。

应显示：

1. 前若干行样本数据。
2. 列名。

#### 数据汇总区

应显示：

1. 总记录数。
2. 有摘要的记录数。
3. 当前激活文件。
4. 当前数据源提示。

#### 文件管理区

每个文件应显示：

1. 文件名。
2. 上传时间。
3. 记录数。
4. 是否当前使用。

用户应能执行的操作：

1. 设为当前使用文件。
2. 删除文件。

### 3.5.5 多文件管理要求

当一个项目下存在多个数据表时，系统必须满足以下要求：

1. 用户能看见项目中所有已上传文件。
2. 用户知道当前筛选、聚类分析或提取所基于的是哪一个文件。
3. 用户能主动切换当前使用的数据源。
4. 用户能删除不再需要的文件。
5. 删除文件时，应清理该文件关联的派生结果，防止结果污染。

### 3.5.6 正确行为要求

1. 当前数据源的概念必须在数据接入页和后续分析页中保持一致。
2. 如果某个文件被删除，而它曾是当前激活文件，系统应自动提醒用户重新选择数据源。
3. 用户在执行筛选前，应清楚知道当前使用的是哪一个文件。

## 3.6 文献筛选页

文献筛选页是系统最复杂、最关键的页面。它不仅负责配置筛选标准，还负责管理 Prompt、版本历史、双语审阅和筛选执行。

页面建议分成 3 个大步骤：

1. 步骤 2.1：初版标准与 AI 扩写。
2. 步骤 2.2：组装并审阅最终 Prompt。
3. 步骤 2.3：执行筛选并导出结果。

## 3.6.1 步骤 2.1：初版标准

### 页面作用

用户在这里定义本轮筛选的核心标准。

### 用户需要填写的内容

1. 综述主题。
2. 研究关键点。
3. 纳入标准草稿。
4. 排除标准草稿。
5. 筛选维度。

### 筛选维度的正确结构

每个维度都必须包含：

1. 系统自动编号：`D1`、`D2`、`D3` ……。
2. 用户填写的维度名称。
3. 用户填写的维度说明。

说明：

1. 维度编号用于 Prompt 组装和结果导出。
2. 结果表中的列名应优先使用维度编号，而不是直接使用维度名称。
3. 用户填写的维度名称和说明可为中文，但进入最终英文 Prompt 时应转为英文。

### 维度增删交互要求

正确交互应为：

1. 每一行维度右侧直接提供 `+` 和 `-` 或删除图标。
2. 按钮与该行输入框应垂直对齐。
3. 不应要求用户通过额外下拉框或单独区域才能继续添加维度。

### 初版标准的填写建议

页面应明确提示：

1. 建议优先使用英文输入。
2. 若使用中文输入，可在 AI 扩写路径中自动转成英文版标准。

## 3.6.2 AI 扩写

### 页面作用

AI 扩写用于把用户的草稿标准扩展为更规范的英文学术筛选标准。

### 用户可选路径

系统应允许两种路径并行存在：

1. 使用 AI 扩写。
2. 跳过 AI 扩写。

不能把“是否使用 AI 扩写”写成用户的强制前置条件。

### 使用 AI 扩写后的正确行为

1. 系统应生成英文版综述主题。
2. 系统应生成更完整的英文纳入标准。
3. 系统应生成更完整的英文排除标准。
4. 系统应生成对各筛选维度的解释规则。

### AI 扩写结果展示要求

1. 扩写结果应显示在单独区域。
2. 用户可以直接编辑扩写后的英文内容。
3. “保存标准”和“修改历史”按钮应位于扩写结果区域下方。
4. 如果扩写失败、没有真正翻译成英文或几乎没有变化，系统应给出明确提示。

## 3.6.3 跳过 AI 扩写

当用户选择跳过 AI 扩写时，正确行为应为：

1. 系统不再显示扩写结果区域。
2. 点击“保存标准”后应自动进入下一步，并完成 Prompt 组装。
3. 页面应直接展示组装好的完整 Prompt，而不是让用户再额外触发一轮无意义操作。

## 3.6.4 标准保存与修改历史

筛选标准在实际研究中通常需要反复修订，因此系统应支持历史管理。

### 建议保留两类历史

1. 标准快照：保存草稿标准、维度和 AI 扩写结果。
2. Prompt 版本：保存组装后的完整 Prompt。

### “修改历史”的作用

1. 回看不同时间点的筛选标准。
2. 恢复某个历史版本。
3. 比较不同版本的 Prompt 或标准。

### 正确行为要求

1. 用户保存标准后，不应丢失当前输入。
2. 恢复历史版本后，应清楚回填到对应编辑区。
3. Prompt 版本保存应独立于草稿标准快照。

## 3.6.5 步骤 2.2：组装并审阅最终 Prompt

### 页面作用

这一步用于把用户标准转换为模型实际执行的最终 Prompt，并让用户完成英文最终确认和双语审阅。

### 组装 Prompt 的正确行为

1. 用户点击“组装最终 Prompt”后，完整 Prompt 应自动显示在下方的大文本框中。
2. 用户不应再额外点击文本框才能看到全部内容。
3. 若用户走的是“跳过 AI 扩写”路径，系统可在保存标准后自动完成这一步。
4. 若用户走的是“AI 扩写”路径，保留显式“组装 Prompt”按钮是合理的。

### Prompt 文本框

作用：

1. 显示最终英文 Prompt。
2. 允许用户继续手工微调。

正确行为：

1. 文本框中应完整显示当前 Prompt。
2. 用户修改后，系统应记住当前编辑内容。
3. 该区域应支持“保存为新版本”。

### 双语审阅

双语审阅是步骤 2 的一部分，不应被拆成一个独立、远离 Prompt 的孤立页面。

建议布局：

1. 完整 Prompt 文本框居中显示。
2. 右上角提供醒目的“双语审阅”按钮。
3. 文本框下方提供“确认开始筛选”按钮。

### 双语审阅的正确结果

1. 成功后应显示完整的中英文对照内容。
2. 不能只显示类似 `{'answer': '...'}` 的字典片段。
3. 中英文对照视图应支持同步滚动。
4. 滚动时标题背景不能透出正文。
5. 翻译成功后，完整 Prompt 文本框可以自动折叠，以便用户优先看到双语对照。

### “保存为新版本”的作用

这个按钮的目标语义应是：

1. 保存当前完整 Prompt 的一个新版本。
2. 不清空当前 Prompt。
3. 不让当前页面内容消失。
4. 便于用户进行 Prompt 迭代。

本 test list 应明确指出，这个按钮是“Prompt 版本管理功能”，不是“清空或重置当前内容”的按钮。

## 3.6.6 步骤 2.3：执行筛选

### 页面作用

在确认最终 Prompt 后，系统按当前数据源逐条执行标题与摘要筛选。

### 数据源选择

在多文件项目中，这一步必须清楚显示：

1. 当前用于筛选的数据文件名称。
2. 当前数据文件的记录数。
3. 如有多个文件，用户可以切换使用的文件。

### 参数设置

通常包括：

1. 批次大小。
2. 速率限制。
3. 温度值。

### 执行过程

系统会：

1. 逐条读取记录的标题和摘要。
2. 将内容填入最终 Prompt。
3. 调用模型。
4. 存储结构化结果。

### 结果展示

结果区应展示：

1. 总论文数。
2. 纳入数。
3. 排除数。
4. 待定数。
5. 置信度分布。
6. 详细结果表。
7. 导出按钮。

### 结果导出要求

结果表应尽可能包含：

1. `Record-id`
2. `Decision`
3. `Confidence`
4. `Reason_Code`
5. `Rationale`
6. 维度列 `D1..Dn`

重点要求：

1. 维度列应使用 `D1..Dn`，而不是直接用中文维度名称。

## 3.7 聚类分析页

## 3.7.1 页面作用

该页面用于对筛选后的文献做主题聚类和主题概览。

## 3.7.2 数据源选择

应支持两种方式：

1. 继承文献筛选结果。
2. 上传新的 CSV/XLSX 数据集。

### 正确行为要求

1. 如果选择“继承筛选结果”，系统应明确说明使用的是哪些筛选结果。
2. 若没有筛选结果，应给出清楚提示，而不是显示原始 key。
3. 中文模式下，诸如 `inherit_screening_label`、`no_screening_results` 等界面词组必须正确翻译显示。

## 3.7.3 BERTopic 的角色说明

本 test list 应明确说明：

1. BERTopic 聚类分析本身不依赖大模型。
2. 聚类主流程依赖文本嵌入、降维和聚类算法。
3. LLM 是可选后处理能力，只用于：
   - 主题命名
   - 主题简介
   - 图表解读

这样用户就能理解：即使没有 LLM，也应能完成聚类；有 LLM 时则可获得更好的主题解释。

## 3.7.4 参数面板

页面应允许用户调整：

1. `random_state`
2. `n_neighbors`
3. `n_components`
4. `min_topic_size`
5. `nr_topics`

并清楚说明：

1. 这些参数影响主题颗粒度和可视化效果。

## 3.7.5 聚类结果展示

应至少包含：

1. 主题概览表。
2. 气泡图。
3. 关键词条形图。
4. 层级图。

## 3.7.6 LLM 后处理

应支持：

1. 为主题自动命名。
2. 生成主题说明简介。
3. 对特定图表生成解释。

### 图表解读的正确行为

1. 图表解读必须与具体图表绑定。
2. 用户对 A 图生成了解读后，再去看 B 图并生成解读，不应覆盖 A 图已有解读。
3. 切换图表后，之前图表的解读仍应可查看。

## 3.7.7 导出结果

导出结果表应保留原始输入列，并新增：

1. `Topic`
2. `Topic_Name`

用户应能：

1. 自定义导出文件名。
2. 下载 CSV 文件。

## 3.8 PDF 提取页

## 3.8.1 页面作用

该页面用于批量上传 PDF，并根据示例模板提取结构化信息。

## 3.8.2 页面流程

标准流程应为：

1. 上传 PDF。
2. 上传示例模板。
3. 预览模板。
4. 保存模板。
5. 设置提取参数。
6. 点击“开始提取”。
7. 查看并导出结果。

## 3.8.3 模板的作用

示例模板用于告诉大模型：

1. 需要提取哪些字段。
2. 这些字段的名称是什么。
3. 结果表应该长成什么样。

这是 PDF 提取页面的关键前置，不应被省略。

## 3.8.4 模板上传与预览要求

正确交互应为：

1. 用户上传模板后，页面先显示模板预览。
2. 模板至少展示前几行样例数据。
3. 用户确认无误后，再点击“保存模板”。

## 3.8.5 开始提取的正确行为

1. 上传模板本身不应自动启动提取。
2. 只有点击“开始提取”按钮后才开始分析。
3. 结果表应按模板字段输出。

## 3.8.6 结果展示

提取结果页面应显示：

1. 文件名。
2. 模板字段内容。
3. 错误信息列。

并支持：

1. 下载 CSV。

## 6. 关键交互与体验要求汇总

这一节是后续开发时的重要对照清单。

### 项目与导航

1. “打开项目”按钮必须直接打开对应项目。
2. 项目内应支持固定顶部导航。
3. 用户下滑后无需回到页面顶部即可切换模块。

### 文献筛选

1. 维度采用 `D1..Dn + 维度名称 + 维度说明` 结构。
2. 维度增删为行内操作。
3. 增删按钮与输入框对齐。
4. AI 扩写是可选项。
5. 跳过 AI 扩写后保存标准应自动进入 Prompt 展示。
6. 使用 AI 扩写时，保存标准按钮应位于扩写结果下方。
7. 组装 Prompt 后应自动完整显示。
8. 双语审阅显示完整内容，不显示字典片段。
9. 双语审阅支持同步滚动。
10. 双语审阅成功后可自动折叠 Prompt。
11. 支持标准历史和 Prompt 版本历史。

### 数据接入

1. 项目内多个数据文件必须可见。
2. 用户必须可删除数据文件。
3. 用户必须知道当前使用的是哪一个文件。
4. 筛选执行前必须能确认数据源。

### BERTopic

1. 聚类主流程不依赖 LLM。
2. LLM 仅用于命名、简介和图表解读。
3. 图表解读与图表一一绑定。
4. 切换图表时不覆盖已有解读。

### PDF 提取

1. 用户先上传模板。
2. 模板先预览后保存。
3. 只有点击“开始提取”才执行提取。

### 国际化

1. 中文模式下，界面不能出现未翻译 key。
2. 诸如 `inherit_screening_label`、`no_screening_results` 这类字符串必须转成正常中文。

### API 配置

1. API Key 应可保存并持久化。
2. API 应尽量支持与模型/供应商绑定。
3. 应支持常见国产模型供应商。

## 7. 常见问题与注意事项

## 7.1 API 相关

1. 如果 AI 扩写、双语审阅、PDF 提取失败，首先检查 API Key、Base URL 和模型名。
2. 若使用国产模型，需确认其接口是否兼容当前调用方式。

## 7.2 文献筛选相关

1. 如果标准是中文，建议通过 AI 扩写得到英文标准后再组装 Prompt。
2. 在多文件项目中，执行筛选前先确认当前数据源。
3. Prompt 每次重要修改后，建议保存为新版本。

## 7.3 BERTopic 相关

1. 聚类结果过少或数据量太小，会导致主题不稳定。
2. 图表解读属于后处理，不影响原始聚类结果。

## 7.4 PDF 提取相关

1. 模板决定结果列结构，模板设计质量直接影响结果可用性。
2. 如果字段名模糊，提取结果会不稳定。
